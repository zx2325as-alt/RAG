# 湖北移动知识库问答系统 - 数据存储与读取技术详解

本文档详细介绍了本系统中关于**文档解析**、**知识库数据存储**、**向量化读取**以及**检索融合**的技术选型、具体细节与设计考量。

---

## 1. 物理文件存储与管理

### 1.1 具体细节
当用户通过前端上传文件（PDF、Word、TXT、图片等）时，后端会根据用户选择的“知识库分类 (db_name)”将文件保存到对应的物理文件夹中。
- **存储路径**: `uploads/{db_name}/{filename}`
- **去重逻辑**: 上传前检查 `document` 数据库表中是否已存在同名且同库的文件，如果已存在且状态为 `completed`，则直接复用，删除临时上传的文件。

### 1.2 为什么这样选择？
- **物理隔离**: 按知识库类别分文件夹存储，不仅便于后续系统维护人员直接查看源文件，也能在极端情况下（如数据库损坏）快速恢复和重建特定库的知识。
- **防止冗余**: 基础的文件名/MD5去重能有效防止用户重复上传相同的运维手册，节省磁盘空间和后续的向量化算力开销。

---

## 2. 关系型数据存储 (SQLite + SQLAlchemy)

### 2.1 具体细节
系统使用 SQLite 作为关系型数据库，主要包含两个核心表：
1. **Document 表**: 记录文档的元数据，包括 `doc_id`、`doc_name`、`db_name`（所属库）、`status`（处理状态）。
2. **Chunk 表**: 记录文档被切分后的具体文本块，包含 `chunk_id`、`doc_id`（外键）、`content`（文本内容）、`chunk_index`。

### 2.2 利用了什么技术？
- **SQLAlchemy ORM**: 提供面向对象的数据操作接口。
- **Flask-SQLAlchemy**: 将 SQLAlchemy 深度集成到 Flask 的应用上下文中。

### 2.3 为什么这样选择？
- **轻量易部署**: 作为一个本地运维知识助手，使用 SQLite 避免了额外部署 MySQL/PostgreSQL 服务的负担，做到开箱即用（Out of the box）。
- **细粒度溯源**: 将 `Chunk` 与 `Document` 建立外键关联。在检索到某个文本块时，可以通过 `doc_id` 精确追溯到它来自哪一本运维手册，这对于给出“参考资料来源”至关重要。

## 3. 向量索引存储 (FAISS)

### 3.1 具体细节
系统将切分好的 `Chunk` 文本通过 Embedding 模型（如 `bge-large-zh`）转化为高维稠密向量，并构建 FAISS 索引。
- **多库隔离**: 内存中维护一个 `self.vector_stores` 字典，键为 `db_name`。物理磁盘上，索引被分别保存在 `faiss_index/{db_name}/index.faiss`。
- **增量更新**: 当有新文件上传时，只将新生成的 Chunks 向量化并调用 `vs.add_documents()` 增量追加到对应库的 FAISS 索引中，然后调用 `vs.save_local()` 覆盖保存。

### 3.2 利用了什么技术？
- **FAISS (Facebook AI Similarity Search)**: 极其高效的本地向量相似度检索库。
- **LangChain**: 提供对 FAISS 的高级封装（`FAISS.load_local`, `FAISS.from_documents`），使得文档元数据（Metadata，如 `chunk_id`, `db_name`）可以和向量一起被存储和召回。

### 3.3 为什么这样选择？
- **性能与成本的平衡**: 相比于部署庞大的 Milvus 或 Qdrant 向量数据库，FAISS 能够以极低的内存开销在本地完成千万级向量的毫秒级检索，非常适合垂直领域的 RAG 系统。
- **元数据绑定**: 借助 LangChain 的封装，检索返回的不仅是相似度得分，还能直接拿到原始文本和对应的 `chunk_id`，为后续的上下文拼接和大模型生成提供直接支撑。

## 4. 关键词倒排索引 (BM25)

### 4.1 具体细节
除了向量检索，系统在内存中为每个数据库并行构建了一套 BM25 词频索引。
- **分词策略**: 不使用传统的 `jieba` 分词，而是直接使用大模型/重排序模型（Reranker）自带的 `tokenizer` 对 Chunk 文本进行 Tokenize。
- **存储结构**: `self.bm25_stores` 字典维护了 `{"db_name": {"bm25": BM25Okapi_Instance, "chunk_map": {chunk_id: Chunk_Object}}}`。

### 4.2 利用了什么技术？
- **rank_bm25 (BM25Okapi)**: 经典的基于 TF-IDF 变种的关键词匹配算法库。

### 4.3 为什么这样选择？
- **弥补向量检索的短板**: 向量模型（Embedding）擅长捕捉语义层面的相似度，但在面对**专有名词、具体的设备型号、精确的错误代码**（如 "OSPF ExStart", "BGP-4", "Error 404"）时往往表现不佳。BM25 基于精确的字词匹配，能够完美兜底这类运维场景下的硬性关键词查询。
- **对齐 Tokenizer**: 使用 Reranker 的 Tokenizer 而不是 Jieba，可以保证分词粒度与后续深度学习模型的感受野完全一致，避免因分词器不同导致的匹配偏差。

## 5. 多路召回与融合读取策略 (RRF + Rerank)

### 5.1 具体细节
当用户发起查询时，系统的数据读取（检索）流程如下：
1. **多库并行召回**: 遍历前端传入的 `db_names` 列表，分别从对应的 FAISS 库提取 Top-K 向量结果，从对应的 BM25 库提取 Top-K 关键词结果。
2. **RRF 排序融合 (Reciprocal Rank Fusion)**: 将来自 FAISS 的距离排序和 BM25 的得分排序，映射为统一的 RRF 积分：`Score = 1 / (k + rank)`。将所有结果打平混合，选出综合排名前 50 的 Chunk。
3. **Cross-Encoder 重排序**: 将这 50 个 Chunk 与原 Query 组成句子对，送入 Cross-Encoder（如 `bge-reranker`）进行深度的交叉注意力计算，得出最终的精准相关性得分。
4. **截断与组装**: 取重排序后的最终 Top-5，拼接成 Prompt 上下文，交给大模型生成最终回答。

### 5.2 利用了什么技术？
- **RRF (Reciprocal Rank Fusion)**: 一种无需训练权重的经典排序融合算法，有效解决不同检索器得分尺度不一致的问题。
- **Cross-Encoder**: 交叉编码器，相较于双塔模型（Bi-Encoder / Embedding），它让 Query 和 Document 在底层网络就开始进行交互（Self-Attention），精度极高。

### 5.3 为什么这样选择？
- **解决“多库”查询的得分不可比问题**: 不同知识库、不同检索算法（L2 距离 vs BM25 分数）的绝对数值完全不具备可比性。RRF 通过**只看排名不看绝对分**的方式，完美实现了跨库、跨算法的公平竞争。
- **精度最大化**: 虽然 Cross-Encoder 计算昂贵（不可预先计算建立索引），但我们将其放置在漏斗的最后一步（仅对 Top-50 重排），在保证系统响应速度（流式秒级响应）的同时，将召回文本的质量提升到了极致，极大地减少了大模型的“幻觉”现象。
