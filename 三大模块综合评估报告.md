## 3. 项目整体判断

### 3.1 总体定位

从代码结构和功能形态来看，该项目不是单纯的 RAG Demo，而是一个已经具备较完整 AI 应用链路雏形的单体平台，核心方向明确：

- 通过知识库与检索增强实现问答系统
- 通过数据准备、持续预训练、教师微调、蒸馏、量化与部署构建模型训练平台
- 通过训练监控、单模型评估、多模型对比构建评估子系统

### 3.2 核心优势

- 三大模块方向清晰，职责边界基本明确
- 问答、训练、评估三条主链路已经具备联动能力
- 项目不是停留在页面层，而是有真实的数据流、模型流、评估流
- 对本地模型生态兼容度较高，支持在线模型、Ollama、vLLM、本地 Hugging Face 目录等

### 3.3 核心短板

- 共享基础设施尚未统一抽象
- 运行时状态较多依赖内存变量和目录扫描
- 训练/部署/评估等重任务缺少统一任务中心
- 超大文件明显，后续维护成本较高
- 评估系统和报告系统还不够成熟

### 3.4 综合结论

该项目当前处于：

**“功能闭环已形成，平台化基础仍需加强”的阶段。**

换句话说：

- 已经“能用”
- 也具备“继续做大的基础”
- 但距离“完备、稳定、可持续演进的平台”还有一段工程化距离

## 4. 系统总架构概览

### 4.1 应用结构

项目采用单体 Flask 架构，入口在：

- `app/main.py`
- `app/__init__.py`

应用初始化时会完成：

- 目录创建
- 日志初始化
- 数据库初始化
- 蓝图注册

其中核心蓝图包括：

- `api_bp`
- `eval_bp`

说明项目虽然是单体应用，但已经把常规业务与评估业务做了初步拆分。

### 4.2 共享基础设施

三个模块共享以下基础设施：

- 配置中心：`Config`
- 数据库：SQLAlchemy
- 文件系统目录：
  - `uploads`
  - `finetuned_models`
  - `finetune_configs`
  - `faiss_index`
  - `logs`
  - `hugface`
- 模型服务：
  - 在线模型
  - Ollama
  - vLLM
  - 本地 HF 模型
- 训练/部署/可视化进程管理：
  - TensorBoard
  - vLLM 进程
  - 微调/CPT 等子进程

### 4.3 三大模块关系

三个模块并不是割裂的，而是串联关系：

1. 问答系统负责知识入库、检索增强、回答生成、反馈回流
2. 模型训练与部署模块负责数据准备、模型训练、蒸馏、量化、服务化部署
3. 评估系统负责训练监控、模型打分、结果对比与导出

形成的整体链路是：

**知识采集 → 问答使用 → 数据回流 → 模型训练 → 模型评估 → 模型部署 → 重新服务问答**

这是该项目最大的结构性价值。

## 5.3 技术实现细节

### 5.3.1 文档入库链路

主流程如下：

1. 前端上传文件
2. 后端写入上传目录
3. 创建文档记录
4. 异步解析文本
5. 清洗与切块
6. Chunk 入库
7. 图谱抽取
8. 更新 FAISS/BM25

该流程体现出项目不是“先把文本存着再说”，而是尽量在文档进入系统时就完成后续检索所需的数据准备。

### 5.3.2 混合检索链路

问答系统的检索不是单一向量召回，而是组合型结构：

- 图谱召回
- 向量召回
- BM25 召回
- 语义去重
- RRF 融合
- reranker 重排

这条链路的技术成熟度明显高于普通 RAG 项目。

### 5.3.3 图谱增强

图谱模块通过 LLM 对文档 chunk 进行实体关系抽取，并写入 Neo4j，用于：

- 1-hop / 2-hop / 3-hop 查询
- 多实体关联查询
- 辅助检索增强

这意味着系统不是简单“文档检索问答”，而是尝试往 Graph RAG 方向延展。

### 5.3.4 回答生成

问答服务会完成：

- 模型初始化
- 上下文构造
- 多来源拼接
- 流式输出
- 查询日志记录

支持：

- DeepSeek
- ChatGPT
- Qwen
- Ollama
- vLLM

模型切换灵活度较高。

## 5.4 业务流程评估

### 5.4.1 用户主流程

1. 新建或选择知识库

## 6.3 技术实现细节

### 6.3.1 数据准备链路

原始数据位于：

- `uploads/datasets/raw`

输出数据位于：

- `uploads/datasets/cleaned`

这一阶段会完成：

- 数据读取
- 去重
- 质量筛选
- 结构化转换

需要注意的是，当前所谓 MinHash 更像一种简化版近似实现，不是严格的数据工程级 MinHash 流水线，因此适合作为原型能力，不适合作为大规模高精度清洗方案。

### 6.3.2 CPT 链路

当前 CPT 不是伪页面，而是真正使用 LLaMA-Factory 执行 `stage: pt` 的持续预训练。

流程是：

1. 读取 cleaned 数据
2. 转成只包含 `text` 的 JSONL
3. 注册数据集描述
4. 生成训练 YAML
5. 拉起 `llamafactory-cli train`
6. 检查完整模型产物
7. 将产物衔接到教师微调

这里最重要的价值是：

**训练链路从“直接 SFT”升级为“领域继续预训练 + 教师微调”的两阶段体系。**

### 6.3.3 教师模型微调链路

教师微调是当前模块中实现最成熟的部分之一，具备：

- 本地模型 / Ollama / vLLM / HF 统一来源兼容
- 训练参数完整配置
- 自动生成训练 YAML
- 自动 merge 产物
- 日志流式回传

这意味着它不仅是“能训练”，而且已经具备一定工程使用价值。

### 6.3.4 蒸馏链路

当前蒸馏模块做了不少工作：

- 加载教师模型
- 生成 logits/软标签
- 生成蒸馏元数据
- 写出蒸馏训练脚本

但问题是：

当前完整自定义蒸馏 Trainer 的执行链路还不算完全打通，整体更像“正在接近完整落地”的状态。

### 6.3.5 量化与部署链路

量化支持 AWQ，部署支持 vLLM OpenAI 兼容 API。

这是一个非常重要的优点，因为说明系统不仅能训练模型，还考虑了实际推理服务化。

## 6.4 业务流程评估

### 6.4.1 训练主流程

用户视角流程：

1. 数据准备
2. CPT
3. 教师微调
4. 学生蒸馏
5. 量化

### 6.4.2 自动衔接能力

- 前后阶段有自动回填

说明该页面不是简单把多个按钮拼起来，而是有真实的流程设计。

## 6.5 优势

## 7.3 技术实现细节

### 7.3.1 训练监控数据来源

监控数据主要来自：

- `trainer_state.json`
- `trainer_log.jsonl`

也就是说，训练监控并不是基于统一任务中心，而是依赖模型目录中的训练产物文件。

这种方式的优点是简单直接，缺点是：

- 可追踪性有限
- 多实例和任务恢复较弱
- 数据口径不够统一

### 7.3.2 评估实现

单模型评估采用本地加载模型推理的方式，逐样本生成，再计算：

- ROUGE-1
- ROUGE-2
- ROUGE-L
- BLEU
- BERTScore F1

这是一个标准的文本生成评估思路，适合当前项目阶段。

### 7.3.3 对比实现

多模型对比本质是：

- 依次运行多个单模型评估
- 聚合指标
- 保存样例
- 展示图表

从工程结构看，已经具备“模型对比平台”的基本雏形。

## 14. 本次 Bug 修复记录

本节用于记录本轮针对实际运行问题所做的修复，保持该报告文件作为统一的分析与修改说明文档持续更新。

### 14.1 Bug1：TensorBoard 启动参数冲突

#### 问题现象

线上点击“实时监控（TensorBoard）”时，后端启动失败，日志中出现：

- `TensorFlow installation not found - running with reduced feature set.`
- `Error: Must not specify both --host and --bind_all.`

其中第一条是 TensorBoard 在缺少 TensorFlow 时的提示，不是导致失败的根因；真正导致失败的是第二条参数冲突。

#### 根因分析

当前后端在启动 TensorBoard 时，同时传递了：

- `--host 0.0.0.0`
- `--bind_all`

这两个参数在 TensorBoard 中不能同时出现，因此子进程直接退出，前端就表现为启动失败。

#### 修改内容

修改文件：

- `app/api/system_routes.py`

修改方式：

- 保留 `--port`
- 当 `TENSORBOARD_HOST == 0.0.0.0` 时，仅追加 `--bind_all`
- 只有在非 `0.0.0.0` 时，才追加 `--host`

#### 为什么这样修改

这样修改可以保持原有“本地/线上都可启动 TensorBoard”的逻辑不变，同时消除参数冲突，兼容当前 TensorBoard 命令行行为。

#### 利用的技术

- 命令参数条件拼装
- 兼容性启动策略

***

### 14.2 Bug2：VL 模型进入纯文本 CPT 训练链路导致 Transformers 架构不识别

#### 问题现象

CPT 训练过程中报错：

- `KeyError: 'qwen2_5_vl'`
- `Transformers does not recognize this architecture`

最终导致：

- `CPT 训练进程异常退出，状态码: 1`

#### 根因分析

当前用户选择的基座模型是：

- `qwen3-vl:2b`

系统原先把它映射成：

- `Qwen/Qwen2.5-VL-3B-Instruct`

但当前 CPT 训练链路处理的是**纯文本语料**，并且服务器上的 `Transformers` 版本对 `qwen2_5_vl` 架构不兼容，因此：

- 从业务上，这条链路本身不需要视觉模型
- 从技术上，当前环境也无法稳定加载该 VL 架构

#### 修改内容

修改文件：

- `app/api/finetune_routes.py`

修改方式：

- 当检测到 `qwen3-vl:2b`、`qwen3_vl_2b`、`qwen2.5-vl:3b` 这类视觉模型名称时
- 不再映射到 `Qwen/Qwen2.5-VL-3B-Instruct`
- 改为自动映射到同规模纯文本基座：
  - `Qwen/Qwen2.5-3B-Instruct`
- 对 Ollama 视觉模型场景，直接跳过本地视觉基座导出流程，避免继续走不兼容路径
- 同时增加日志提示，明确说明这是因为“当前训练链路仅处理纯文本语料”

#### 为什么这样修改

这样改有两个核心原因：

1. **保持原有训练逻辑不变**
   - 仍然是当前 CPT/SFT 流程
   - 仍然是文本训练
   - 只是把不兼容的视觉基座映射为同规模文本基座
2. **保证 CPT 功能真正可运行**
   - 与其让当前环境在 `Transformers` 架构识别阶段报错退出
   - 不如在业务入口层直接选择与文本训练链路匹配的基座模型

#### 利用的技术

- 基于模型命名规则的兼容性分流
- 文本训练链路与视觉训练链路隔离
- 运行时自动回退映射策略

### 14.3 Bug3：`rag_system.log` 路径迁移到项目根目录 `logs`

#### 问题现象

当前日志文件路径为：

- 启动时检测旧路径：
  - `app/logs/rag_system.log`

#### 为什么这样修改

这样改可以保证：

- 日志路径统一
- 不依赖当前运行目录
- 兼容已有旧日志文件
- 不破坏现有日志初始化逻辑

## 15. 本次工具调用能力优化记录

本节记录问答系统页面中“工具调用”能力的本轮优化，目标是在不改变其他业务逻辑的前提下，让工具调用从演示式实现升级为可实际运行、可继承上下文、可稳定自动调用的实现。

### 15.1 问题概述

原有工具调用能力存在以下几个核心问题：

- 工具集合偏演示化，`query_metrics` 使用的是模拟数据，不是真实系统状态
- 通用工具 Schema 较弱，参数描述不够清晰，不利于模型稳定地产生正确函数参数
- 工具调用链路依赖 AgentExecutor，不同 LangChain 版本兼容性一般，失败后容易整体降级
- 工具模式下多轮对话历史没有真正持久化，导致后续追问的上下文继承不稳定
- 前端虽然有工具开关，但后端没有真正形成“优先工具 -> 工具结果回注 -> 最终回答”的稳固流程
- 当前问答页的会话标识没有真正传递到后端，历史上下文主要依赖 `remote_addr`，多轮对话隔离性较差

### 15.2 修改内容

#### 15.2.1 工具从 Demo 改为真实可用

修改文件：

- `app/services/agent_tools.py`

本次将工具集重构为更贴近现有系统真实能力的工具：

- `search_knowledge_base`
  - 直接调用现有知识库检索链路
  - 返回真实的文档片段、文档名、知识库名、得分
- `get_document_details`
  - 可按 `doc_id` 或 `doc_name` 查询数据库中的文档与 chunk 内容
  - 用于深挖某一篇文档的具体证据
- `query_metrics`
  - 不再返回 mock 数据
  - 改为读取当前真实系统信息，包括：
    - 文档数
    - chunk 数
    - 当前模型
    - 训练任务状态
    - 部署状态
- `execute_shell_command`
  - 改为只允许只读型诊断命令
  - 采用更严格的安全限制和结果结构化输出
- `query_api_endpoint`
  - 保留，但限制为 `http/https`，并输出结构化结果

#### 15.2.2 优化 Function Calling Schema 设计

本次为工具增加了更明确的参数模型和字段说明，包括：

- 字段含义
- 默认值
- 可选范围
- 使用场景

这样做的目的，是让模型在调用工具时能更稳定地产生正确参数，而不是只凭函数名猜测。

#### 15.2.3 优化工具自动调用流程

修改文件：

- `app/services/qa_service.py`

原有方案主要依赖 LangChain 的 AgentExecutor 封装，现在调整为：

- 直接使用 `bind_tools` 构建可调用工具的 LLM
- 通过手动循环管理：
  - LLM 输出工具调用
  - 执行对应工具
  - 将工具结果以 `ToolMessage` 回注
  - 再次让模型决定是否继续调用工具
  - 最终生成答案

这类实现的优点是：

- 对当前版本更可控
- 更容易观察每一步工具调用
- 出问题时更容易排查
- 不依赖某一个 Agent 工厂函数是否存在

#### 15.2.4 优化多轮对话上下文继承

修改文件：

- `app/services/qa_service.py`
- `app/api/llm_routes.py`
- `app/static/js/main.js`

本次做了三层优化：

1. 问题重构不再只在“关闭工具模式”时生效  
   现在工具模式下也会继承历史上下文并做问题补全，这样“它/这个/刚才那个问题”这类追问也能更准确进入工具链

2. 工具模式下的最终问答结果会手动写回会话历史  
   原先工具模式下历史没有真正持久化，多轮继承效果会越来越差；现在已经补上

3. 前端正式引入会话 ID 并传给后端  
   不再只依赖 `remote_addr`，页面级会话上下文隔离更稳定  
   清空对话时也会重置新的会话 ID，使前后端上下文一致

#### 15.2.5 优化缓存与多轮上下文兼容

原有缓存 key 主要依赖问题文本，容易导致多轮追问在不同上下文下误命中缓存。

本次调整为：

- 缓存 key 纳入模型、知识库、工具开关、历史摘要信息
- 仅在“无历史上下文 + 非工具模式”时优先使用缓存

这样可以避免多轮追问时因为缓存导致答案和当前语境不一致。

#### 15.2.6 优化前端工具调用体验

修改文件：

- `app/static/js/main.js`

优化内容包括：

- 会话 ID 初始化与持久化
- 请求时把 `session_id` 带给 `/query` 与 `/feedback`
- 清空对话时重置会话 ID
- 工具调用流程完成后，思考过程区域能够正确结束并折叠，不再长期停留在“处理中”

### 15.3 为什么这样修改

这次修改的核心原则是：

- 不改动现有问答主流程
- 不推翻现有知识库检索逻辑
- 不改动页面整体结构
- 只把原本偏 Demo 的工具调用能力，增强成真正可运行的工程能力

也就是说：

- 检索、知识库、引用、思考过程、评分等原有逻辑保持不变
- 优化重点只落在“工具定义、工具 Schema、工具自动调用、多轮继承、自动流程稳定性”这几个点上

### 15.4 利用的技术

- LangChain 工具绑定 `bind_tools`
- `ToolMessage` 回注式函数调用循环
- Pydantic 参数 Schema 建模
- 基于会话 ID 的多轮上下文隔离
- 工具调用结果结构化 JSON 输出
- 基于历史摘要的缓存隔离策略

### 15.5 本轮修改后的直接效果

本轮修改后，问答页中的工具调用能力将具备以下提升：

- 工具不再主要依赖 mock 数据，能够调用现有系统真实能力
- 模型更容易稳定地产生正确的工具参数
- 多轮追问时，上下文继承更稳定
- 工具模式下的对话历史会真正参与后续轮次
- 工具自动调用链路更可控、更容易排障
- 清空对话后，前后端上下文状态能保持一致
